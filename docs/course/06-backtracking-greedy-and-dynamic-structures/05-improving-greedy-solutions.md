# Improving Greedy Solutions

## A first solution is not the end of the design process

A greedy construction often gives a fast, feasible solution. Even when it is not guaranteed optimal, it can be a useful starting point. The next question is how to improve it or how to replace it with a method that provides a stronger guarantee.

The syllabus distinguishes **systematic** and **heuristic** approaches. The distinction concerns how alternatives are examined and what can be concluded from the search.

## Systematic improvement of an existing solution

A systematic improvement method explores changes according to a complete rule within a precisely defined search region.

Examples include:

- testing every one-swap neighbour of a current ordering;
- testing every segment reversal in a route;
- exhaustive search around an initial solution;
- branch and bound that uses the greedy solution as the current incumbent.

A systematic procedure supports a precise statement such as:

```text
No one-swap neighbour is better than the current solution.
```

That is a guarantee relative to the chosen neighbourhood. It is not automatically a proof of global optimality.

### Branch and bound and the role of a greedy incumbent

A greedy solution can provide an initial bound for branch and bound. Any partial branch that cannot beat the current greedy solution may be pruned. The exact method is still branch and bound; the greedy solution supplies a useful incumbent rather than being gradually transformed into the exact answer.

## Exact alternatives are not merely greedy improvements

Some problems admit an exact method based on a different algorithmic structure.

**Dynamic programming** is the principal example. It solves and combines overlapping subproblems under an appropriate recurrence. It should not be described as a generic procedure that improves a greedy result. A greedy solution may be used as a comparison baseline or as a bound, but dynamic programming is a separate design paradigm with its own state definition, recurrence, and correctness argument.

The same distinction applies to exhaustive search: it may start from a greedy bound, but its guarantee comes from systematic coverage of the relevant search space.

## Heuristic improvement

A heuristic uses practical guidance to find promising changes quickly. It may stop before all alternatives are checked and usually does not prove global optimality.

Common heuristic choices include:

- examine promising candidates first;
- accept only improvements;
- allow occasional worsening moves to escape a local optimum;
- restart from several initial solutions;
- limit the number of iterations or elapsed time.

## Local search example

Suppose a route is represented by an ordering of points. A simple move reverses one segment when that shortens the route.

```python
def improve_route(route: list[int], distance) -> list[int]:
    best = route.copy()
    improved = True

    while improved:
        improved = False

        for left in range(1, len(best) - 2):
            for right in range(left + 1, len(best) - 1):
                candidate = best[:left] + list(reversed(best[left:right])) + best[right:]

                if route_length(candidate, distance) < route_length(best, distance):
                    best = candidate
                    improved = True
                    break

            if improved:
                break

    return best
```

The move set is systematic: candidate segment reversals are generated by a defined rule. The acceptance policy is **first improvement** because the scan stops after the first better candidate.

A **best-improvement** variant would evaluate the complete neighbourhood, select the best improving move, and apply it only after the scan finishes. Best improvement usually performs more work per iteration and can follow a different path through the solution space.

Both variants remain heuristic as global solvers. A route with no improving reversal may still be far from globally optimal.

## Local optima

A **local optimum** is no worse than its neighbours under a chosen move rule. Change the neighbourhood and the meaning of local optimum changes.

An improvement algorithm must therefore state:

- the representation of a solution;
- the neighbourhood relation;
- the order in which neighbours are examined;
- whether the policy is first improvement or best improvement;
- the evaluation function;
- the acceptance rule;
- the stopping condition.

## Measuring progress

An improvement loop needs a progress argument. Common measures include:

- strictly decreasing cost;
- strictly increasing score;
- a bounded iteration counter;
- a finite set of states combined with a rule that prevents revisiting.

Without such a condition, the procedure may oscillate.

## Evaluating a heuristic honestly

A heuristic should be assessed by:

- solution quality;
- running time;
- memory use;
- sensitivity to the starting solution;
- consistency across test instances;
- comparison with known optimal answers on small inputs.

The result should not be described as optimal unless a proof or exhaustive verification supports that claim.
